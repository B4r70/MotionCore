### [L1-002] `HealthMetricCalcEngine` ist `class : ObservableObject` und nimmt `AppSettings` im Init

**Severity:** рҹҹ  High
**Kategorie:** CalcEngine-Reinheit
**Datei:** Services/Calculation/HealthMetricCalcEngine.swift:13вҖ“38

- Verwirrt das Pattern: andere CalcEngines (`StatisticCalcEngine`, `CoreSessionCalcEngine`, `XPCalcEngine` вҖҰ) sind `struct` вҖ” diese sticht heraus
- Erschwert Test/Preview, weil immer ein `AppSettings`-Singleton mitgegeben werden muss
- `ObservableObject` ohne `@Published`-Properties ist Cargo-Cult вҖ” die Klasse profitiert nicht von der KonformitГӨt

**Empfohlene Korrektur:**
Auf `struct` umstellen, Initializer mit den fГјnf konkret benГ¶tigten Werten (`birthday`, `age`, `gender`, `bodyHeight`, `activityLevel`) statt mit `AppSettings`. `calculateTodayCalorieBalance` nimmt drei `Int?`-Werte (consumed/basal/active) statt `HealthKitManager`.     }


### [L1-008] `ExerciseSet` snapshotet Exercise-IdentitГӨt als `persistentModelID.hashValue.description`

**Severity:** рҹ”ҙ Critical
**Kategorie:** Persistenz / DatenintegritГӨt
**Datei:** Models/Core/ExerciseSet.swift:104вҖ“118
**Verwandte Findings:** вҖ”

**Fundstelle:**
```swift

    self.init(
        exerciseName: exercise.name,
        exerciseNameSnapshot: exercise.name,
        exerciseUUIDSnapshot: exercise.persistentModelID.hashValue.description,
        // ...
    )
    self.exercise = exercise
}
```

**Problem:**
`exerciseUUIDSnapshot` ist die *stabile IdentitГӨts-BrГјcke* zwischen ExerciseSet und Exercise (siehe `repairSnapshotsOnLaunch` in `BaseView.swift:497вҖ“522` und `groupKey`-Verwendung in `ExerciseSet.swift:267вҖ“271`). Hier wird sie aber aus `exercise.persistentModelID.hashValue.description` gebaut. `Hashable.hashValue` ist in Swift dokumentiert als **nicht stabil zwischen ProgrammlГӨufen** ("`hashValue` is randomly seeded each program execution"). Selbst innerhalb desselben Laufs ist `PersistentIdentifier.hashValue` nicht garantiert stabil (Apple gibt keine StabilitГӨtsgarantie fГјr `hashValue`). Das bedeutet: Sets, die mit diesem Convenience-Init angelegt werden, haben einen exerciseUUIDSnapshot, der bei jedem App-Restart anders sein kann вҖ” jedenfalls aber **kein gГјltiger UUID-String** ist.

`BaseView.repairSnapshotsOnLaunch` lГ¶scht ja gerade alle nicht-UUID-Strings (`UUID(uuidString: uuid) == nil`) вҖ” also wird dieser Snapshot beim nГӨchsten Launch sofort wieder geleert, womit der Set seine Exercise-IdentitГӨt verliert. Die Tatsache, dass dieser Repair-Code Гјberhaupt existiert, deutet darauf hin, dass das Problem in der Praxis bereits auftritt.

**Auswirkung:**
- Statistiken/PRs/Heatmaps gruppieren Гјber `groupKey` (= `exerciseUUIDSnapshot` falls nicht leer, sonst Name) вҖ” verlorener Snapshot bedeutet unstabile Gruppierung Гјber Sessions hinweg
- Smart-Progression / Plan-Update arbeiten mit `groupKey` вҶ’ Treffer auf Гңbungen verfehlt
- Repair-Code in `BaseView` muss die Sets bei jedem Launch wieder aufrГӨumen, was den Snapshot effektiv lГ¶scht

**Empfohlene Korrektur:**
`Exercise` hat bereits ein `apiID: UUID?`-Feld. Falls `apiID` gesetzt ist, sollte das die Snapshot-Quelle sein. Andernfalls braucht `Exercise` eine eigene stabile UUID-Property (analog zu `sessionUUID`/`planUUID`/`setUUID`), die im `init` generiert wird.

**Konkreter Fix:**
```swift

// Custom-Exercises (apiID == nil) mГјssen einen Fallback bekommen вҖ” z.B. neue UUID-Property in Exercise.

// In Exercise.swift:

// (CloudKit UUID Default Bug aus CLAUDE.md beachten)

// In ExerciseSet.swift:
convenience init(from exercise: Exercise, setNumber: Int = 1, weight: Double = 0, reps: Int = 0) {
    self.init(
        exerciseName: exercise.name,
        exerciseNameSnapshot: exercise.name,
        exerciseUUIDSnapshot: (exercise.apiID ?? exercise.exerciseUUID).uuidString,
        // ...
    )
    self.exercise = exercise
}
```

**Aufwand:** ~30 Min (plus Migration / `deduplicateAllSyncUUIDs`-Aufruf wegen CloudKit-Default-UUID-Bug)
