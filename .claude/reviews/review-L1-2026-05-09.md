Cluster, das zusammen passt: **L1-003 bis L1-005** (überflüssige `import SwiftUI` in `Models/Core/TrainingPlan.swift`, `Models/Core/AppSettings.swift`, `Services/Calculation/StatisticCalcEngine.swift`) — alle drei sind ein One-Liner-Removal pro Datei.

### [L1-001] Preview-Container registriert nur 2 von 14 SwiftData-Typen

**Status:** ✅ Umgesetzt am 2026-05-14. Schema auf alle 14 Typen aus `MotionCoreApp.appSchema` erweitert.

**Severity:** 🟠 High
**Kategorie:** Persistenz / Preview-Infrastruktur
**Datei:** Examples/MockData/PreviewModelContainer.swift:16–33
**Verwandte Findings:** —

**Fundstelle:**
```swift
#if DEBUG
@MainActor
struct PreviewData {
    static let sharedContainer: ModelContainer = {
        let config = ModelConfiguration(isStoredInMemoryOnly: true)
        let container = try! ModelContainer(for: CardioSession.self,
                                            BodyMeasurement.self,
                                            configurations: config)

        let context = container.mainContext
        CardioSession.previewMockData.forEach { context.insert($0) }

    }()
}
#endif
```

**Problem:**
`MotionCoreApp.appSchema` enthält 14 Modelle (`StrengthSession`, `OutdoorSession`, `ExerciseSet`, `ExerciseMetrics`, `ExerciseRating`, `Exercise`, `TrainingPlan`, `Studio`, `StudioEquipment`, `ExerciseProgressionState`, `SessionReadiness`, `HealthBaseline` etc.). Der Preview-Container registriert nur `CardioSession` und `BodyMeasurement`. Jede Preview, die eine `StrengthSession`, einen `TrainingPlan` oder ein `Exercise` instanziiert, oder ein View, das per `@Query` etwas ausserhalb dieser zwei Typen sucht, läuft gegen einen Container, der das Modell gar nicht kennt — Preview-Crash zur Laufzeit.

**Auswirkung:**
- Previews vieler Views (alle Strength-, Plan-, Exercise-, Studio-, Heatmap-, Readiness-Views) brechen still oder verhalten sich inkonsistent
- Entwickler können nicht zuverlässig "Cmd+B + Preview" als Validierungspfad nutzen
- Refactorings, die Modelle anfassen, fallen erst bei vollständigem App-Run auf

**Empfohlene Korrektur:**
`PreviewData.sharedContainer` muss dasselbe Schema wie `MotionCoreApp.appSchema` benutzen, idealerweise indem das Schema selbst in eine zentrale Konstante extrahiert wird, die beide nutzen. Der Schema-Drift zwischen App- und Preview-Container ist Architektur-relevant: In-Memory-Persistenz ist OK, aber die Modell-Liste muss synchron sein.

**Konkreter Fix:**

```swift
                Exercise.self,

                HealthBaseline.self,
                BodyMeasurement.self,
            configurations: config

        let context = container.mainContext
        CardioSession.previewMockData.forEach { context.insert($0) }

    }()
}
#endif
```

### [L1-002] `HealthMetricCalcEngine` ist `class : ObservableObject` und nimmt `AppSettings` im Init

**Severity:** 🟠 High
**Kategorie:** CalcEngine-Reinheit
**Datei:** Services/Calculation/HealthMetricCalcEngine.swift:13–38
**Verwandte Findings:** [L1-003], [L1-005]

**Fundstelle:**
```swift
import Foundation
import Combine
import SwiftUI

class HealthMetricCalcEngine: ObservableObject {

    let allWorkouts: [CardioSession]

    private let userBirthdayDate: Date
    // ...

    init(workouts: [CardioSession], settings: AppSettings) {
        self.allWorkouts = workouts
        self.userBirthdayDate = settings.userBirthdayDate
        // ...
    }
}
```

**Problem:**
Laut Swift-Standards-Skill und CLAUDE.md sind CalcEngines "pure structs, no state, no side effects". Diese Engine ist eine `class : ObservableObject` und akzeptiert eine ganze `AppSettings`-Instanz im Initializer (statt der konkret benötigten Werte). Sie stellt zudem eine Methode `calculateTodayCalorieBalance(from healthKit: HealthKitManager)` bereit — Engine reicht in einen Singleton hinein, statt den benötigten Wert als Input zu erhalten.

**Auswirkung:**
- Verwirrt das Pattern: andere CalcEngines (`StatisticCalcEngine`, `CoreSessionCalcEngine`, `XPCalcEngine` …) sind `struct` — diese sticht heraus
- Erschwert Test/Preview, weil immer ein `AppSettings`-Singleton mitgegeben werden muss
- `ObservableObject` ohne `@Published`-Properties ist Cargo-Cult — die Klasse profitiert nicht von der Konformität

**Empfohlene Korrektur:**
Auf `struct` umstellen, Initializer mit den fünf konkret benötigten Werten (`birthday`, `age`, `gender`, `bodyHeight`, `activityLevel`) statt mit `AppSettings`. `calculateTodayCalorieBalance` nimmt drei `Int?`-Werte (consumed/basal/active) statt `HealthKitManager`. Engine wird damit pure und in Previews testbar.

**Konkreter Fix:**
```swift
import Foundation

struct HealthMetricCalcEngine {
    let allWorkouts: [CardioSession]
    let userBirthdayDate: Date
    let userAge: Int
    let userGender: Gender
    let userBodyHeight: Int
    let userActivityLevel: UserActivityLevel

    init(
        workouts: [CardioSession],
        birthday: Date,
        age: Int,
        gender: Gender,
        bodyHeight: Int,
        activityLevel: UserActivityLevel
    ) {
        self.allWorkouts = workouts
        self.userBirthdayDate = birthday
        self.userAge = age
        self.userGender = gender
        self.userBodyHeight = bodyHeight
        self.userActivityLevel = activityLevel
    }

    // calculateTodayCalorieBalance(consumed:basal:active:) statt HealthKitManager
    func calculateTodayCalorieBalance(
        consumed: Int?, basal: Int?, active: Int?
    ) -> CalorieBalance? {
        guard let consumed, let basal, let active else { return nil }
        // ... bestehende Logik
    }
}
```
Aufrufer (View) übernimmt das Mapping `AppSettings` → konkrete Werte und `HealthKitManager` → drei `Int?`.

**Aufwand:** ~30 Min (mehrere Aufrufstellen)
**Risiko:** Mittel (Aufrufer müssen mitgezogen werden)
**Diskussion erwünscht:** Nein

### [L1-006] `SupabaseFullBackupService` ist 785 Zeilen — God-Service in der Sync-Pipeline

**Severity:** 🟡 Medium
**Kategorie:** God-Object / Modul-Grenzen
**Datei:** Services/Database/Remote/Session/SupabaseFullBackupService.swift:1–785
**Verwandte Findings:** —

**Fundstelle:**
```swift
@MainActor
final class SupabaseFullBackupService: ObservableObject {
    // 13 private uploadAll…(context:) Methoden:
    //   uploadAllTrainingPlans, uploadAllStudios, uploadAllProgressionStates,
    //   uploadAllExerciseMeta, uploadAllReadinessAndBaselines,
    //   uploadAllStrengthSessions, uploadAllCardioSessions,
    //   uploadAllOutdoorSessions, uploadAllBodyMeasurements,
    //   uploadAllTemplateSets, uploadAllExerciseSets, …
}
```

**Problem:**
Der Service vereint Backup-Orchestrierung (`runFullBackup`) mit 13 typ-spezifischen Upload-Methoden in einer einzigen Datei. Mit 785 Zeilen liegt er deutlich über der Hard-Stop-Grenze (800) und nahe dran. Architektonisch ist er ein klassischer God-Service: jede neue Modell-Klasse zwingt zu einer Änderung in dieser Datei. Pro Modell gibt es jeweils einen DTO-Mapping-Block, der dem Mapping in `SupabaseSessionService` ähnelt, aber nicht geteilt wird.

**Auswirkung:**
- File-Size-Compliance grenzwertig (Zielwert 400, Hard-Stop 800)
- Hohe Kopplung: ein Bug im StrengthSession-Mapping zwingt zur Änderung im selben File wie ein Bug im Studio-Mapping
- Erschwert Code-Reviews, weil jedes Backup-Feature im selben Diff landet

**Empfohlene Korrektur:**
Pro Domain einen kleinen `…BackupUploader`-Service extrahieren (`StrengthBackupUploader`, `PlanBackupUploader`, `StudioBackupUploader` …). `SupabaseFullBackupService` wird dann nur noch Orchestrator und Progress-Reporter, mit ~150 Zeilen. DTO-Mapping kann dabei mit `SupabaseSessionService` gebündelt werden, indem die Single-Item-Upload-Methoden dort wiederverwendet werden.

**Konkreter Fix:**

```swift
// … etc.

enum StrengthBackupUploader {

        sessions: [StrengthSession],

    ) async throws -> (sessions: Int, sets: Int) {
        // Logik aus uploadAllStrengthSessions() hierher
    }
}

@MainActor
final class SupabaseFullBackupService: ObservableObject {
    // dünner Orchestrator, ruft die Uploader sequentiell
}
```

**Aufwand:** mehrere Stunden
**Risiko:** Mittel (Backup ist datenkritisch — Tests vor/nach durch Side-by-Side-Diff sichern)
**Diskussion erwünscht:** Ja
**Begründung Diskussion:** Trade-off: aktueller Stand ist linear lesbar (alle Uploads in einem Scroll), Aufsplittung erhöht die Anzahl Files, gewinnt aber Wartbarkeit. Bartoszs Präferenz hier ist nicht offensichtlich.

### [L1-007] `BaseView` mischt UI-Komposition mit App-Lifecycle-Orchestrierung

**Severity:** 🟡 Medium
**Kategorie:** Cross-Layer / God-Object
**Datei:** Views/Root/View/BaseView.swift:1–537
**Verwandte Findings:** —

**Fundstelle:**
```swift
struct BaseView: View {
    // 14 @State-Variablen
    // @Query für Widget-Snapshot
    // 5 Tabs als verschachtelte NavigationStacks
    // 4 Sheet-Modifier
    // .onAppear → repairSnapshotsOnLaunch + SupabaseResyncService + WidgetSnapshotPublisher + MuscleRecoverySnapshot
    // .onChange(scenePhase) → noch einmal das Gleiche + planImportManager.poll
    // private func handleSessionRestoration / restoreStrengthSession / startStrengthSession
    // private func triggerDailyMuscleRecoverySnapshotIfNeeded (mit MuscleRecoveryCalcEngine.analyze + Supabase-Upload)
    // private func repairSnapshotsOnLaunch (SwiftData Migration-Reparatur)
}
```

**Problem:**
`BaseView` ist Tab-Container *und* App-Lifecycle-Coordinator gleichzeitig. Sie:
1. Stellt die TabView dar.
2. Repariert SwiftData-Snapshots beim Launch (`repairSnapshotsOnLaunch`).
3. Triggert tägliche Supabase-Uploads (`triggerDailyMuscleRecoverySnapshotIfNeeded`).
4. Triggert `SupabaseResyncService` und `WidgetSnapshotPublisher` bei Launch & Foreground.
5. Restauriert Sessions per Notification-Handler.
6. Manages Plan-Import-Polling.

Damit verstößt sie gegen das L1-Prinzip "Views koordinieren, keine Business-Logik" und gegen "Coordinator/Orchestrator-Logik gehört nicht in den Root-View".

**Auswirkung:**
- 537 Zeilen in einem Root-View, schwer überblickbar
- Side-Effects (Supabase-Upload) verstecken sich in `.onAppear`, wo man sie nicht sucht
- Tests/Previews sind praktisch unmöglich, weil zu viele Abhängigkeiten
- Wenn ein Lifecycle-Schritt umzieht (z.B. zu einem `AppLifecycleCoordinator`), muss man durch die ganze View pflügen

**Empfohlene Korrektur:**
Lifecycle-Logik in einen dedizierten Coordinator/ViewModifier auslagern, ähnlich wie `handleSessionLifecycle()` in `ActiveSessionManager.swift` bereits zeigt. `BaseView` bleibt dünner Tab-Container.

**Konkreter Fix:**
```swift
// Neue Datei: Services/Lifecycle/AppLaunchCoordinator.swift
@MainActor
final class AppLaunchCoordinator {
    func runOnLaunch(context: ModelContext) async { /* repair + resync + widget + recovery */ }
    func runOnForeground(context: ModelContext) async { /* resync + widget + recovery + planImport */ }
}

// In BaseView:
.task {
    await launchCoordinator.runOnLaunch(context: context)
}
.onChange(of: scenePhase) { _, new in
    guard new == .active else { return }
    Task { await launchCoordinator.runOnForeground(context: context) }
}
```

**Aufwand:** ~1h
**Risiko:** Mittel (Reihenfolge der Side-Effects ist heute teilweise verzögert per `DispatchQueue.main.asyncAfter` — muss erhalten bleiben)
**Diskussion erwünscht:** Ja
**Begründung Diskussion:** Die zeitliche Reihenfolge (`.now() + 1.0` für CloudKit-Boot) ist heute in `BaseView.onAppear` versteckt. Wenn man das in einen Coordinator zieht, muss diese Verzögerung explizit dokumentiert werden — sonst geht das Wissen verloren.

### [L1-008] `ExerciseSet` snapshotet Exercise-Identität als `persistentModelID.hashValue.description`

**Severity:** 🔴 Critical
**Kategorie:** Persistenz / Datenintegrität
**Datei:** Models/Core/ExerciseSet.swift:104–118
**Verwandte Findings:** —

**Fundstelle:**
```swift
convenience init(from exercise: Exercise, setNumber: Int = 1, weight: Double = 0, reps: Int = 0) {
    self.init(
        exerciseName: exercise.name,
        exerciseNameSnapshot: exercise.name,
        exerciseUUIDSnapshot: exercise.persistentModelID.hashValue.description,
        // ...
    )
    self.exercise = exercise
}
```

**Problem:**
`exerciseUUIDSnapshot` ist die *stabile Identitäts-Brücke* zwischen ExerciseSet und Exercise (siehe `repairSnapshotsOnLaunch` in `BaseView.swift:497–522` und `groupKey`-Verwendung in `ExerciseSet.swift:267–271`). Hier wird sie aber aus `exercise.persistentModelID.hashValue.description` gebaut. `Hashable.hashValue` ist in Swift dokumentiert als **nicht stabil zwischen Programmläufen** ("`hashValue` is randomly seeded each program execution"). Selbst innerhalb desselben Laufs ist `PersistentIdentifier.hashValue` nicht garantiert stabil (Apple gibt keine Stabilitätsgarantie für `hashValue`). Das bedeutet: Sets, die mit diesem Convenience-Init angelegt werden, haben einen exerciseUUIDSnapshot, der bei jedem App-Restart anders sein kann — jedenfalls aber **kein gültiger UUID-String** ist.

`BaseView.repairSnapshotsOnLaunch` löscht ja gerade alle nicht-UUID-Strings (`UUID(uuidString: uuid) == nil`) — also wird dieser Snapshot beim nächsten Launch sofort wieder geleert, womit der Set seine Exercise-Identität verliert. Die Tatsache, dass dieser Repair-Code überhaupt existiert, deutet darauf hin, dass das Problem in der Praxis bereits auftritt.

**Auswirkung:**
- Statistiken/PRs/Heatmaps gruppieren über `groupKey` (= `exerciseUUIDSnapshot` falls nicht leer, sonst Name) — verlorener Snapshot bedeutet unstabile Gruppierung über Sessions hinweg
- Smart-Progression / Plan-Update arbeiten mit `groupKey` → Treffer auf Übungen verfehlt
- Repair-Code in `BaseView` muss die Sets bei jedem Launch wieder aufräumen, was den Snapshot effektiv löscht

**Empfohlene Korrektur:**
`Exercise` hat bereits ein `apiID: UUID?`-Feld. Falls `apiID` gesetzt ist, sollte das die Snapshot-Quelle sein. Andernfalls braucht `Exercise` eine eigene stabile UUID-Property (analog zu `sessionUUID`/`planUUID`/`setUUID`), die im `init` generiert wird.

**Konkreter Fix:**
```swift
// Variante A (klein): solange Exercise eine apiID hat, diese nutzen.
// Custom-Exercises (apiID == nil) müssen einen Fallback bekommen — z.B. neue UUID-Property in Exercise.

// In Exercise.swift:
var exerciseUUID: UUID = UUID()  // CloudKit-Default — siehe deduplicateAllSyncUUIDs
// (CloudKit UUID Default Bug aus CLAUDE.md beachten)

// In ExerciseSet.swift:
convenience init(from exercise: Exercise, setNumber: Int = 1, weight: Double = 0, reps: Int = 0) {
    self.init(
        exerciseName: exercise.name,
        exerciseNameSnapshot: exercise.name,
        exerciseUUIDSnapshot: (exercise.apiID ?? exercise.exerciseUUID).uuidString,
        // ...
    )
    self.exercise = exercise
}
```

**Aufwand:** ~30 Min (plus Migration / `deduplicateAllSyncUUIDs`-Aufruf wegen CloudKit-Default-UUID-Bug)
**Risiko:** Hoch (wirkt auf Statistik-Gruppierung, PR-Detection, Plan-Update)
**Diskussion erwünscht:** Ja
**Begründung Diskussion:** Es ist möglich, dass dieser Convenience-Init in der Produktion gar nicht mehr aufgerufen wird (alle Aufrufe gehen über Plan-Templates, die `cloneForSession()` nutzen, das einen bestehenden Snapshot weiterreicht). In dem Fall genügt es, den Convenience-Init als `@available(*, deprecated)` zu markieren oder zu löschen. Vor dem Fix sollte geprüft werden, wo `ExerciseSet(from: exercise)` tatsächlich noch aufgerufen wird.

### [L1-009] `PlanUpdateApplicator` liegt in `Services/Calculation/`, mutiert aber `ModelContext`

**Severity:** 🟡 Medium
**Kategorie:** Modul-Grenzen / Naming
**Datei:** Services/Calculation/PlanUpdateApplicator.swift:14–123

```swift
struct PlanUpdateApplicator {

        context: ModelContext,
        sourceSessionUUID: String? = nil

        // context.insert(...) / context.delete(...) / plan.removeTemplateSets(...)
    }

```

**Problem:**
`Services/Calculation/` ist laut Architektur der Ort für *pure CalcEngines* (siehe SKILL.md: "CalcEngines in Services/Calculation/, pure and stateless"). `PlanUpdateApplicator` ist genau das Gegenteil: er mutiert `ModelContext` (`insert`, `delete`) und `TrainingPlan`-Felder (`lastUpdatedFromSession`, `lastUpdateSourceSessionUUID`). @StateObject private var backupService = SupabaseFullBackupService()

### [L1-012] `SupabaseClient.shared` ruft `fatalError` auf fehlende Konfiguration

**Severity:** 🟠 High
**Kategorie:** Bootstrap / Robustheit
**Datei:** Services/Database/Remote/Core/SupabaseClient.swift:21–28
**Verwandte Findings:** —

**Fundstelle:**
```swift
private init() {
    do {
        self.baseURL = try SupabaseConfig.url
        self.anonKey = try SupabaseConfig.anonKey
    } catch {
        fatalError("Supabase Konfiguration fehlt: \(error)")
    }
}
```

**Problem:**
`SupabaseClient.shared` wird beim ersten Zugriff lazy initialisiert. Wenn die Konfiguration fehlt (z.B. weil der `Secrets.plist` in einem Build-Configuration nicht eingebunden ist), crasht die App **bei jedem ersten Zugriff** auf irgendeinen Supabase-Service. Da Supabase laut CLAUDE.md "sekundärer Mirror" ist und CloudKit primäre Persistenz, sollte das Fehlen der Supabase-Konfiguration **kein Crash-Grund** sein — die App muss ohne Supabase weiterlaufen können (graceful degradation). `MotionCoreApp.swift:33` definiert sogar explizit `requireCloudKit = false` als Fallback-Strategie — das gleiche Prinzip fehlt für Supabase.

**Auswirkung:**
- Hard-Crash bei kaputter/fehlender Supabase-Config statt sauberer Fehlermeldung
- Verstößt gegen das dokumentierte Prinzip "Supabase ist sekundär — Fehler werden geloggt, nicht weitergereicht" (z.B. in `SupabaseSessionService.swift:13`)
- Fehleranalyse erschwert: ein fataler Crash beim App-Start gibt keinen Hinweis darauf, *welcher* Subservice betroffen ist

**Empfohlene Korrektur:**
`SupabaseClient` darf nicht crashen. Stattdessen `nil`-baseURL oder `disabled`-Flag — alle Methoden sollten in dem Fall kontrolliert `throws` werden. Aufrufer (alle Sync-Services) propagieren den Fehler bereits in ein `do/catch` und loggen ihn — das Gesamt-Pattern ist also schon dafür ausgelegt.

**Konkreter Fix:**
```swift
final class SupabaseClient {
    static let shared = SupabaseClient()

    private let baseURL: URL?
    private let anonKey: String?

    private init() {
        self.baseURL = (try? SupabaseConfig.url)
        self.anonKey = (try? SupabaseConfig.anonKey)
        if baseURL == nil || anonKey == nil {
            print("⚠️ SupabaseClient: Konfiguration fehlt — Supabase deaktiviert")
        }
    }

    private func makeRequest(url: URL, method: String) throws -> URLRequest {
        guard let anonKey else { throw SupabaseError.notConfigured }
        // …
    }

    // get/post/upsert/delete: alle starten mit
    //     guard let baseURL else { throw SupabaseError.notConfigured }
}
```
`SupabaseError.notConfigured` als neuer Case ergänzen.
