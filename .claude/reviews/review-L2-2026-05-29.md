### Top Themes (what runs through the review?)
1. **Stored-property default vs. init-default divergence** — affects 2 findings. SwiftData/CloudKit hydrates a field-absent record from the *stored-property* default, never the init default. Where the two disagree on a *valid* enum case (not masked by the `?? fallback` getter), migrated or cross-device-synced records silently get a different value than every code path produces. Verified across all nine persisted enum-Raw defaults; exactly two diverge (`ExerciseSet.isCompleted`, `TrainingPlan.planTypeRaw`).
2. **File-size hard-stop** — affects 1 finding. The exercise seeder is ~1196 lines, well past the 800-line hard-stop (static data, but the rule is unconditional).
3. **CloudKit-default discipline is otherwise excellent** — affects 2 (positive) observations. Every `@Model` gives all stored properties defaults, every enum is persisted as a Raw field with a typed computed accessor, and the UUID-dedup coverage is complete.

### Top 3 Wins (what is done well?)
1. **Complete UUID-dedup coverage.** `SupabaseFullBackupService.deduplicateAllSyncUUIDs` (Services, out of scope) covers every model carrying a defaulted UUID identity — `TrainingPlan`, all three sessions, `ExerciseSet`, `ExerciseRating`, `Studio`, `StudioEquipment`, `ExerciseProgressionState`, `SessionReadiness`, `HealthBaseline`, `BodyMeasurement` — plus a separate `deduplicateExerciseUUIDs`. The CLAUDE.md CloudKit-UUID-default gotcha is handled systematically rather than ad hoc.
2. **Consistent and correct enum-as-Raw persistence pattern.** Every model that needs an enum stores a `…Raw` String/Int with a CloudKit default and exposes a typed computed property with a `?? fallback` getter. Verified that each stored default equals its initializer default (`Intensity.none == 0`, `SetKind.work == "work"`, `OutdoorActivity.cycling`, `WeatherCondition.unknown`, `TrainingProgram.random`, `StudioEquipmentType.machine`) — the pattern is sound; only `TrainingPlan.planTypeRaw` (L2-002) breaks it.
3. **Snapshot-resolution robustness in `ExerciseSet`.** `resolveSnapshot(existing:exercise:)` (`ExerciseSet.swift:336`) prioritises an existing valid UUID, falls back to `apiID`/`exerciseUUID`, and finally to name-based `groupKey` — preventing the empty-snapshot drift that would otherwise break groupKey-based grouping across all clone/create paths.

### Recommendation
Tackle **L2-001** first — it is the only one with a real data-correctness consequence and the fix is a one-character change. **L2-002** (TrainingPlan default) is the same class of bug; fix both together since they are one mental model. **L2-003** (seeder size) is a maintainability item that can wait and should be discussed (split strategy). The remaining findings are Info/structural and need no urgent action.

## Findings

### ExerciseSet.swift

### [L2-001] `ExerciseSet.isCompleted` stored default is `true` while every init path uses `false`

**Severity:** 🟠 High
**Category:** Persistence / CloudKit default
**File:** Models/Core/ExerciseSet.swift:59
**Related Findings:** [L2-002]
**Touches Locked Area:** No

**Location:**
```swift
var isCompleted: Bool = true                 // Satz abgeschlossen?
```
```swift
// init, line 183:
isCompleted: Bool = false,
```

**Problem:**
The stored-property default (`true`) and the designated-initializer default (`false`) disagree. SwiftData/CloudKit uses the *stored-property* default when materialising a record whose `isCompleted` column is absent — e.g. a schema migration that adds the attribute, or a cross-device sync where the field never arrived. Those records become `isCompleted = true` even though no code path ever creates a completed set by default (`cloneForSession` even forces `false` explicitly).

**Impact:**
- Legacy/migrated sets can appear completed without ever being trained.
- `StrengthSession.completedSets`, `.progress`, and `.allSetsCompleted` (all derived from `isCompleted`) over-report.
- `totalVolume` is unaffected, but session-progress UI and any "session done" gating would be wrong for affected records.

**Recommended Correction:**
Make the stored default match the only value the app actually produces (`false`). The init default and `cloneForSession` already assume `false`, so this aligns the persistence default with intent. No data backfill is needed for correctly-written records (they always carry an explicit value); the change only affects future field-absent hydration.

**Concrete Fix:**
```swift
var isCompleted: Bool = false                // Satz abgeschlossen?
```

**Effort:** <5 min
**Risk:** Low
**Discussion Wanted:** Yes
**Discussion Reason:** Confirm the `true` default was not a deliberate choice for some legacy import path. If any historical migration relied on field-absent sets defaulting to completed, flipping it would change how those records read — but no such path is visible in L2.

### TrainingPlan.swift

### [L2-002] `TrainingPlan.planTypeRaw` stored default `"cardio"` diverges from init default `.mixed`

**Severity:** 🟡 Medium
**Category:** Persistence / CloudKit default
**File:** Models/Core/TrainingPlan.swift:37
**Related Findings:** [L2-001]
**Touches Locked Area:** No

**Location:**
```swift
var planTypeRaw: String = "cardio"      // "cardio", "strength", "outdoor", "mixed"
```
```swift
// init, line 340 + 347:
planType: PlanType = .mixed,
...
self.planTypeRaw = planType.rawValue
```

**Problem:**
The stored default is `"cardio"`; the init default is `.mixed`. `"cardio"` is a *valid* `PlanType` case (confirmed in `TrainingTypes.swift:19`), so the typed getter (`PlanType(rawValue:) ?? .mixed`) does **not** mask it — a record hydrated without the field reads as `.cardio`, not `.mixed`. Code-created plans always carry an explicit value, so the divergence only surfaces on field-absent CloudKit hydration / migration.

**Impact:**
- A migrated or sync-incomplete plan with no `planTypeRaw` reads as a Cardio plan instead of the intended Mixed default.
- Plan-type-driven filtering/icons (`PlanType.icon`, `.color`) would mislabel such plans.

**Recommended Correction:**
Align the stored default with the init default. Set the stored default to `"mixed"` so field-absent hydration matches the value the initializer would have produced.

**Concrete Fix:**
```swift
var planTypeRaw: String = "mixed"       // "cardio", "strength", "outdoor", "mixed"
```

**Effort:** <5 min
**Risk:** Low
**Discussion Wanted:** Yes
**Discussion Reason:** Two valid resolutions — change the stored default to `"mixed"` (matches init), or change the init default to `.cardio` (matches storage). Which is the true intended default for a plan with no type is a product decision.

### ExerciseSeeder.swift

### [L2-003] `ExerciseSeeder` is ~1196 lines — past the 800-line hard-stop

**Severity:** 🟡 Medium
**Category:** Maintainability / file size
**File:** Models/Seeder/ExerciseSeeder.swift:1–1196
**Related Findings:** —
**Touches Locked Area:** No

**Location:**
```swift
private static func createAllExercises() -> [Exercise] {
    var exercises: [Exercise] = []
    var sortIndex = 0
    // ... ~1050 lines of exercises.append(Exercise(...)) ...
    return exercises
}
```

**Problem:**
The file is roughly 1196 lines, exceeding the 800-line hard-stop. The bulk is a single `createAllExercises()` factory building ~55 `Exercise` instances inline. It is static seed data rather than logic, but the size rule is unconditional and the monolithic factory is hard to diff and review per-change.

**Impact:**
- Editing or adding one exercise forces scrolling through a 1000-line method.
- Merge conflicts on seed changes are noisy.
- Reviewing a single exercise's parameters in isolation is hard.

**Recommended Correction:**
Split the seed data out of the orchestration logic. Move the per-muscle-group blocks into separate factory files (e.g. `ExerciseSeeder+Chest.swift`, `+Back.swift`, …) returning `[Exercise]`, and keep `seedMissing`/`upsertAll`/`reseed` in the main file. Alternatively, move the data to a bundled JSON resource decoded at seed time — that also removes the `sortIndex` closure-mutation pattern.

**Concrete Fix:**
```swift
// ExerciseSeeder.swift keeps orchestration only:
private static func createAllExercises() -> [Exercise] {
    reindexSortOrder(
        chestExercises()
        + backExercises()
        + shoulderExercises()
        + armExercises()
        + legExercises()
        + gluteExercises()
        + coreExercises()
    )
}

// ExerciseSeeder+Chest.swift
extension ExerciseSeeder {
    static func chestExercises() -> [Exercise] { /* ... */ }
}
```

**Effort:** ~1h
**Risk:** Low
**Discussion Wanted:** Yes
**Discussion Reason:** Per-file split vs. JSON-resource migration is a trade-off (compile-time safety + refactor cost vs. data/code separation). Worth a quick decision before investing the hour.

### ExerciseSet.swift

### [L2-004] `calculatedRIR` does not account for the `rpe == 0` "not captured" sentinel

**Severity:** ⚪ Info
**Category:** Domain semantics
**File:** Models/Core/ExerciseSet.swift:132–135
**Related Findings:** —
**Touches Locked Area:** No

**Location:**
```swift
// RIR berechnet aus RPE
var calculatedRIR: Int {
    max(0, 10 - rpe)
}
```

**Problem:**
Per CLAUDE.md, `rpe == 0` means "not captured" (disambiguated by `rpeRecorded`), **not** RIR 10. `calculatedRIR` maps `rpe == 0` to `10 - 0 = 10` (RIR 10) without consulting `rpeRecorded`. Whether this is a bug depends entirely on the call sites, which could not be searched this session (Grep unavailable), so impact is unverified.

**Impact:**
- *If* any consumer reads `calculatedRIR` without first checking `rpeRecorded`/`rpe > 0`, an uncaptured set would be treated as a maximally-easy RIR-10 set. (Unverified — call sites not inspected.)
- `ProgressionCalcEngine.hasRIRData` is documented to guard with `rpe > 0`, which would make this property safe in the progression path specifically.

**Recommended Correction:**
Either make the property optional to encode "no data" honestly, or leave it and rely on caller-side guards (the existing `hasRIRData` pattern). No change recommended until a call-site audit confirms an unguarded consumer exists.

**Concrete Fix:**
```swift
// Option (if an unguarded caller is found):
var calculatedRIR: Int? {
    guard rpeRecorded, rpe > 0 else { return nil }
    return max(0, 10 - rpe)
}
```

**Effort:** ~10 min
**Risk:** Medium
**Discussion Wanted:** Yes
**Discussion Reason:** Impact unverified (no Grep this session). Changing the return type would ripple to call sites; leaving it relies on every caller guarding. Needs a call-site audit before acting.

### AppSettings.swift

### [L2-006] `AppSettings` mixes 40+ unrelated `@Published`/UserDefaults settings in one observable

**Severity:** 🟡 Medium
**Category:** Maintainability / state surface
**File:** Models/Core/AppSettings.swift:18–435
**Related Findings:** —
**Touches Locked Area:** No

**Location:**
```swift
class AppSettings: ObservableObject {
    static let shared = AppSettings()
    @Published var showAnimatedBlob: Bool { didSet { ... } }
    // ... ~30 more @Published, each a UserDefaults didSet ...
    @Published var bodyMeasurementThighMode: BodyMeasurementSideMode = .singleSide { didSet { ... } }
```

**Problem:**
`AppSettings` is a single global `ObservableObject` aggregating display, workout, user, weekly-goal, smart-plan-update, e-bike, health, debug, and body-measurement settings (~435 lines, ~30 `@Published`). Every property change publishes to the same object, so any view observing `AppSettings.shared` re-evaluates on *any* setting change. The file is under the warn threshold but the single-object state surface is broad.

**Impact:**
- Coarse change-notification: a view that only reads `appTheme` re-renders when, say, `eBikeKilometers` changes.
- Settings file is a frequent merge-conflict locus as unrelated features grow it.

**Recommended Correction:**
Consider splitting into focused observables (`DisplaySettings`, `WorkoutSettings`, `EBikeProfile`, `HealthSettings`) behind a façade, or moving rarely-changing groups (e-bike profile) out of the hot observable. This is a structural trade-off, not a defect — flagged for discussion.

**Concrete Fix:**
```swift
// Sketch — façade keeps call sites stable while narrowing observation:
final class AppSettings: ObservableObject {
    static let shared = AppSettings()
    let display = DisplaySettings()
    let workout = WorkoutSettings()
    let eBike   = EBikeProfile()
    // views observe the sub-object they actually use
}
```

**Effort:** several hours
**Risk:** Medium
**Discussion Wanted:** Yes
**Discussion Reason:** A global settings singleton is a legitimate design choice for a single-user app; splitting it touches many call sites for a re-render benefit that may not matter at this app's scale. Worth a deliberate decision rather than a reflexive refactor.
